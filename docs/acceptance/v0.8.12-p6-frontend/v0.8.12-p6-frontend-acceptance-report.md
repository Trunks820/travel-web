## 1. Implemented scope

### 1.1 无固定时间 (optional time preferences)

- `InputPage` time preference state now defaults to empty (was `09:00`/`21:00`).
- Submit body includes `daily_start` / `daily_end` **only when the user filled
  them**, each side independently. Empty preference submits neither field, and
  no default clock value is ever synthesized.
- `MorePreferences` shows `未设置，默认无固定时间` in the collapsed-state hint,
  a per-state explainer line (both / start-only / end-only / none), a
  `清除，恢复无固定时间` reset button, and re-editing restores the empty state
  from stored form data (verified by automation).
- Optional preset chips (朝九晚九 / 睡到自然醒 / 只管晚上) fill the inputs only
  on explicit user click — they are not hidden defaults.

### 1.2 停留时段 (public periods)

- Deleted the synthesized 09:00-cumulative leisure timeline
  (`utils/schedule.ts` `computeSchedule`). The file now only maps backend
  schedule data for display.
- `Timeline` renders `place.schedule.period` as a badge: morning→上午,
  afternoon→下午, evening→傍晚, night→晚上. No clock time is derived from
  place order; the old arrive/leave column is gone.

### 1.3 精确时间 (governed exact time)

- `governedExactTime()` renders exact time **only when**
  `schedule.exact_time_source` is one of
  `reservation | event | transport | verified_venue_rule` **and**
  `exact_start`/`exact_end` are valid `HH:MM`. One-sided anchors render as
  `HH:MM 起` / `HH:MM 前`; full ranges as `HH:MM - HH:MM`, with a source label
  (预约/活动/交通/场馆规则) beside the time.
- Anything without a governed source renders no exact time.

### 1.4 Day 与地点文案

- Day `narrative` renders once at the top of the timeline as the day overview.
- `place.activity_note` renders inside its own place card (1.5); 1.4 records
  fall back to the existing `brief`.
- `activity_note` is never merged into commute cards; transit lines, stops,
  transfers and minutes render only in the commute section.

### 1.5 通勤信息

- `commute_legs` + `transit_steps` rendering preserved (structured line/stop/
  transfer/minutes when provider detail exists; backend `transit_summary`
  fallback text when degraded). The frontend infers no transit facts and does
  not parse commute info out of narrative/activity_note.

### 1.6 Schema 1.4 / 1.5 compatibility

- `types/trip.ts`: additive optional fields only — `TripPlace.activity_note?`,
  `TripPlace.schedule?` (`PlaceScheduleInfo`), `TripResult.time_preferences?`,
  `TripResult.must_include?` (`TripMustInclude`, 5-status enum from backend).
- 1.4 records open unchanged: no fabricated period/activity_note/exact time,
  no must-include card, no `无固定时间` claim (missing field ≠ user chose no
  window; header shows the time-pref chip only for schema ≥ 1.5 via
  `isSchema15()`), transit steps keep rendering.

### 1.7 must-include 状态

- New `MustIncludeCard` (left column of detail page): `scheduled`→已安排,
  `not_scheduled`→未能安排 + backend reason verbatim, `cross_city`→不在本城市,
  `recorded_unmatched`→未匹配到地点, `recorded_candidate`→已记录. Unscheduled
  count badge on the card header. Failures are never hidden or rendered as
  scheduled.

### 1.8 UX fixes bundled in P6 (user-reported)

- Date range picker required two clicks to lock the departure date: the
  open-effect depended on `startDate` and reset in-progress selection state on
  the first click. Fixed by initializing panel state once in `openPanel()`.
  Now: first click = departure (live hint `M/D 出发 · 再点一天作为返程`),
  second click = return, auto-close; hover previews the range; clicking an
  earlier day second auto-swaps into a valid range.
