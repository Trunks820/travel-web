# Product Scope

---

## 1. MVP Pages (v1.0-web)

4 个核心页面，形成最小闭环：

### Page 01: Input — 旅行需求输入

**用户目标**：告诉系统去哪、玩几天、什么偏好。

| 表单字段 | 映射后端字段 | 必填 | 备注 |
|----------|-------------|------|------|
| 目的地 | `trip_request.to_city` | 是 | 支持城市级选择 |
| 游玩天数 | `trip_request.days` | 是 | 1-7 天 |
| 人数 | `trip_request.people_count` | 是 | 默认 1 |
| 旅行偏好 | `trip_request.preferences` | 否 | 标签多选 |
| 旅行节奏 | `trip_request.preferences` | 否 | 轻松/适中 |
| 希望避开 | `trip_request.avoid` | 否 | 标签多选 |
| 补充需求 | `trip_request.notes` | 否 | 自由文本 |

**节奏映射规则**：
- 轻松 → preferences += ["轻松"]
- 适中 → 不写入（默认行为）
- 特种兵/快慢结合 → 第一版不提供

**第一版移除的字段**：
| 字段 | 原因 |
|------|------|
| 出发城市 | 不影响路线规划，展示会暗示支持跨城交通 |
| 出发日期 | v0.7 才做天气，展示会造成错误预期 |
| 预算金额 | 无可靠价格数据源 |

---

### Page 02: Planning — 生成等待

**用户目标**：知道系统在忙什么、还要多久。

展示 4 步进度条：
```
[●] 正在理解旅行需求     (ANALYZING)
[●] 正在筛选地点并规划路线 (PLANNING)
[○] 正在生成旅行方案      (COMPOSING)
[○] 正在校验并整理方案    (FINALIZING)
```

行为：
- 每 2s 轮询 `GET /api/trip/jobs/{job_id}`
- 高亮当前步骤，已完成步骤打勾
- 失败时展示友好错误 + 重试按钮
- 成功后自动跳转结果页

---

### Page 03: Result — 方案总览

**用户目标**：快速对比两套方案，选一个看详情。

每套方案展示：
- 方案标题
- 简短摘要（1-2 句话）
- 标签（如"轻松"、"美食"、"citywalk"）
- 行程节奏状态（RELAXED / MODERATE）
- 总通勤时间

点击方案卡片 → 进入详情页

---

### Page 04: Plan Detail — 方案详情

**用户目标**：逐日查看行程安排 + 地图路线。

**桌面端布局**：左侧行程卡片 / 右侧地图
**移动端布局**：上方地图 / 下方行程卡片滚动

每日卡片包含：
- Day N 标题
- 地点列表（名称、类别、角色标记）
- 通勤段信息（交通方式、时长、距离）
- 当天通勤总结
- 每日叙述文本

地图联动：
- 切换 Day → 地图聚焦当天 POI
- Marker 编号对应行程顺序
- 虚线 Polyline 示意连接
- 标注"路线顺序示意图"

## 3. Non-Goals (v1.0)

明确不做：

- ❌ 消费预算金额展示（无数据源支撑）
- ❌ 用户注册/登录
- ❌ 历史攻略列表
- ❌ 分享功能
- ❌ 地点详情页
- ❌ 真实道路路线（只做示意连线）
- ❌ 出发日期/天气关联
- ❌ 跨城交通
- ❌ "特种兵"/"快慢结合"节奏
- ❌ 国际化 (i18n)
- ❌ SSR / SEO
- ❌ PWA 离线缓存
- ❌ 小程序（Web 先行，后续评估 Taro 迁移）
